# mcp_servers/transactions/

Second downstream MCP server. Wraps the transactions mock API (US-012) and
sits behind the MCP gateway (US-007). Three tools — `get_transactions`,
`get_counterparties`, `flag_velocity_anomalies` — all read-only, all
deterministic.

**Refactored in US-015** to use the shared `mcp_servers/_common.py`
factory; the JSON-RPC + PASETO pipeline is no longer duplicated here.

## Endpoints

| Method | Path             | Purpose                                                                                                              |
| ------ | ---------------- | -------------------------------------------------------------------------------------------------------------------- |
| `POST` | `/transactions`  | MCP JSON-RPC 2.0. Supports `tools/list` and `tools/call`. Requires a valid service-to-service PASETO bearer header. |
| `GET`  | `/healthz`       | Liveness probe.                                                                                                      |

The path is `/transactions` because the gateway POSTs to
`f"{downstream_url}/{server}"` and `SERVER_NAME = "transactions"`. Don't
rename the path without updating both `SERVER_NAME` and the gateway's
downstream registry.

## Tools (contract for US-017 analyze-transactions SKILL.md)

| Tool                      | Required args  | Optional args        | Returns                                                                                                       |
| ------------------------- | -------------- | -------------------- | ------------------------------------------------------------------------------------------------------------- |
| `get_transactions`        | `customer_id`  | `scenario`, `limit`  | `{customer_id, scenario, transactions: [...]}`                                                               |
| `get_counterparties`      | `customer_id`  | `scenario`           | `{customer_id, scenario, counterparties: [...]}`                                                             |
| `flag_velocity_anomalies` | `customer_id`  | `scenario`           | `{customer_id, scenario, transaction_count, inbound_count, structuring_candidate_count, cross_border_count, distinct_counterparty_countries, flags: [...]}` |

Tool input schemas are generated by FastMCP from the python signatures. Don't
change a tool's parameter set without updating
`test_tools_list_schemas_match_analyze_transactions_skill_contract` AND the
US-017 SKILL.md frontmatter when it lands.

## Env vars (production entry point `build_default_app()`)

- `TRANSACTIONS_MCP_PUBLIC_KEY` — PEM path with the service PASETO public
  key. The matching private key lives on the gateway as
  `MCP_GATEWAY_SERVICE_PRIVATE_KEY`. **Same keypair as customer_data uses** —
  the gateway holds the private half once and every downstream MCP server
  verifies against the same public half. Don't issue per-server service
  keypairs without coordinating the gateway's mint path first.
- `TRANSACTIONS_API_URL` — Base URL of the transactions mock API (defaults to
  `http://localhost:8003`).

## Conventions

- Service PASETO validation runs **before** anything else. Same five
  failure modes as customer_data: missing / wrong-scheme / garbage /
  expired / wrong-keypair → 401 + JSON-RPC error body.
- `httpx.AsyncClient` is **injectable**. Tests stitch the mock app
  in-process via `httpx.ASGITransport`.
- FastMCP is used **as a tool registry only**. `tool.run(args)` is the
  execution entry point — don't mount FastMCP's built-in HTTP transport
  because we need PASETO validation in front of any tool call.

## Pitfalls

- `FastMCP.get_tool(name)` returns `None` on miss — it does NOT raise.
- `tool.run(arguments)` is awaitable — don't `asyncio.run()` it inside a
  request handler.
- The mock API returns HTTP 400 for unknown `?scenario=` values. The server
  forwards as `upstream_status=400` in `error.data`. Don't smuggle it into
  a 500 — the eval harness will conflate the cases.
- `limit` is an optional int on `get_transactions`. The FastMCP-generated
  schema makes it optional, but if the SKILL.md ever needs to enforce it
  per-tool the schema is the source of truth — change the python signature
  and the schema test in lockstep.
