# mcp_servers/kyc/

Third downstream MCP server. Wraps the kyc mock API (US-013) and sits behind
the MCP gateway (US-007). Three tools — `get_kyc_record`, `get_document`,
`get_ubo_tree` — all read-only, all deterministic. Same shape as
`mcp_servers/customer_data/` (US-009) and `mcp_servers/transactions/`
(US-012) on purpose — third copy of the duplicated pattern.

## Endpoints

| Method | Path     | Purpose                                                                                                              |
| ------ | -------- | -------------------------------------------------------------------------------------------------------------------- |
| `POST` | `/kyc`   | MCP JSON-RPC 2.0. Supports `tools/list` and `tools/call`. Requires a valid service-to-service PASETO bearer header. |
| `GET`  | `/healthz` | Liveness probe.                                                                                                    |

The path is `/kyc` because the gateway POSTs to `f"{downstream_url}/{server}"`
and `SERVER_NAME = "kyc"`. Don't rename the path without updating both
`SERVER_NAME` and the gateway's downstream registry.

## Tools (contract for the future check-kyc / draft-narrative SKILL.md files)

| Tool             | Required args                       | Optional args | Returns                                                                                                                                                                                |
| ---------------- | ----------------------------------- | ------------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `get_kyc_record` | `customer_id`                       | `scenario`    | `{customer_id, full_name, dob, ssn_last4, id_document_type, id_document_number, issuer_country, verification_method, verified_at_year, kyc_status, pep_flag, sanctions_match, entity_type, inconsistencies: [...]}` |
| `get_document`   | `customer_id`, `document_id`        | `scenario`    | `{document_id, customer_id, kind, issuer_country, expiry_year, verification_method, on_file}`                                                                                          |
| `get_ubo_tree`   | `customer_id`                       | `scenario`    | `{customer_id, entity_type, owners: [...], flags: [...]}`                                                                                                                              |

Tool input schemas are generated by FastMCP from the python signatures. Don't
change a tool's parameter set without updating
`test_tools_list_schemas_match_kyc_contract` in lockstep.

## Env vars (production entry point `build_default_app()`)

- `KYC_MCP_PUBLIC_KEY` — PEM path with the service PASETO public key. The
  matching private key lives on the gateway as
  `MCP_GATEWAY_SERVICE_PRIVATE_KEY`. **Same keypair as customer_data and
  transactions use** — the gateway holds the private half once and every
  downstream MCP server verifies against the same public half.
- `KYC_API_URL` — Base URL of the kyc mock API (defaults to
  `http://localhost:8006`).

## Conventions

- Service PASETO validation runs **before** anything else. Same five failure
  modes as customer_data + transactions: missing / wrong-scheme / garbage /
  expired / wrong-keypair → 401 + JSON-RPC error body.
- `httpx.AsyncClient` is **injectable**. Tests stitch the mock app in-process
  via `httpx.ASGITransport`.
- FastMCP is used **as a tool registry only**. `tool.run(args)` is the
  execution entry point — don't mount FastMCP's built-in HTTP transport
  because we need PASETO validation in front of any tool call.

## Pitfalls

- **US-015 consolidation done.** The shared pipeline now lives in
  `mcp_servers/_common.py` and this server's `create_app` is a thin shell
  that calls `create_jsonrpc_app(...)`.
- `FastMCP.get_tool(name)` returns `None` on miss — it does NOT raise.
- `tool.run(arguments)` is awaitable — don't `asyncio.run()` it inside a
  request handler.
- The mock API returns HTTP 400 for unknown `?scenario=` values and HTTP 404
  for unknown `document_id`. Both surface via `httpx.HTTPStatusError` →
  passthrough with `upstream_status`/`upstream_body` in `error.data`. Don't
  conflate the 400 and 404 into a 500.
- `get_document` takes a `document_id` arg in addition to `customer_id` — the
  only tool in the kyc surface that does. Required by FastMCP schema; the
  document_id must match the IDs the mock returns for that customer+scenario.
