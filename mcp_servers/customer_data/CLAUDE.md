# mcp_servers/customer_data/

The first downstream MCP server. Wraps the customer_data mock API (US-008) and
sits behind the MCP gateway (US-007). Three tools — `get_customer`,
`list_accounts`, `get_device_history` — all read-only, all deterministic.

**Refactored in US-015** to use the shared `mcp_servers/_common.py`
factory; this file is now a thin shell that declares constants + the tool
registry + a `create_app()` that delegates to `create_jsonrpc_app(...)`.
The JSON-RPC + PASETO pipeline is no longer duplicated here.

## Endpoints

| Method | Path              | Purpose                                                                                                                    |
| ------ | ----------------- | -------------------------------------------------------------------------------------------------------------------------- |
| `POST` | `/customer_data`  | MCP JSON-RPC 2.0. Supports `tools/list` and `tools/call`. Requires a valid service-to-service PASETO bearer header.        |
| `GET`  | `/healthz`        | Liveness probe.                                                                                                            |

The path is `/customer_data` because the gateway POSTs to
`f"{downstream_url}/{server}"` and the server name is `customer_data`. If you
ever fork this server for a new MCP server (transactions, kyc, etc.), the URL
suffix must match the `SERVER_NAME` constant.

## Tools (contract for US-010 orchestrator)

| Tool                  | Required args  | Optional args | Returns                                                                                  |
| --------------------- | -------------- | ------------- | ---------------------------------------------------------------------------------------- |
| `get_customer`        | `customer_id`  | `scenario`    | `{customer_id, full_name, dob, country, kyc_status, pep, risk_score, flags, scenario, …}` |
| `list_accounts`       | `customer_id`  | `scenario`    | `{customer_id, scenario, accounts: [{account_id, type, currency, balance, …}, …]}`        |
| `get_device_history`  | `customer_id`  | `scenario`    | `{customer_id, scenario, devices: [{device_id, os, last_login_country, suspicious, …}, …]}` |

Tool input schemas are generated by FastMCP from the python signatures —
**don't** change a tool's parameter set without updating the
`test_tools_list_schemas_match_orchestrator_skill_contract` test AND any
SKILL.md frontmatter that lists this server's tools.

## Env vars (production entry point `build_default_app()`)

- `CUSTOMER_DATA_MCP_PUBLIC_KEY` — PEM path with the service PASETO public key.
  The matching private key lives on the gateway as
  `MCP_GATEWAY_SERVICE_PRIVATE_KEY` — same keypair, two sides.
- `CUSTOMER_DATA_API_URL` — Base URL of the customer_data mock API (defaults
  to `http://localhost:8001`).

## Conventions

- Service PASETO validation runs **before** anything else. No JSON-RPC method
  is dispatched without a valid bearer token. Expired/garbage/wrong-keypair
  tokens → 401 with a JSON-RPC error body (no leaks of the failure reason
  beyond the public taxonomy of "expired" / "invalid").
- The `httpx.AsyncClient` to the upstream mock is **injectable**. Tests use
  `httpx.ASGITransport(app=create_mock_app())` so the entire stack
  (gateway → server → mock) runs in-process without sockets. The same pattern
  will be reused for transactions/kyc/sanctions/osint/case_actions servers
  (US-012 .. US-016) — lift it if a third server lands.
- FastMCP is used purely as a **tool registry**. We don't mount its built-in
  HTTP transport because we need to enforce service PASETO ourselves before
  any tool runs. The FastMCP `FunctionTool.run(args)` call still gives us the
  MCP-shaped `ToolResult` (text content + structured content) so the wire
  contract matches the spec.

## Pitfalls

- `FastMCP.get_tool(name)` returns the tool **or `None`** — it does NOT raise
  on missing names. Check for None explicitly.
- `FastMCP.list_tools(run_middleware=False)` skips the middleware chain. Use
  `run_middleware=False` for `tools/list` so an audit-or-whatever middleware
  added later doesn't accidentally rewrite the contract a client sees.
- The mock API returns HTTP 400 for unknown `?scenario=` values. The server
  surfaces this via `httpx.HTTPStatusError` → 4xx passthrough with
  `upstream_status` / `upstream_body` in `error.data`. Don't smuggle the 400
  into a 500 — the eval harness will conflate the cases.
- `tool.run(arguments)` is the FastMCP execution entry point — it runs the
  registered coroutine in the current event loop. Don't `asyncio.run()` it
  inside a request handler; `await` it.
- Tool parameter validation is FastMCP's job. Missing required args produce
  an internal error inside `tool.run()`; we currently treat it as a 500. If
  that's wrong for the eval scorers, add an explicit shape check before
  `tool.run()` is called.
